# SMB 直连评估（任务 9）

日期：2026-04-06

## 结论

- 结论：`SMB 全功能直连（扫描 + 播放 + 缓存）当前不可直接上线`。
- 可上线范围（建议）：`WebDAV 继续作为主方案`，SMB 保留在二期。
- 可灰度的低风险项：若后续要试点 SMB，可先做“`仅播放 URL`”能力，不做 SMB 扫描和离线缓存。

## PoC 结果

本仓库已有二进制级证据显示 FFmpeg/Libavformat 编译时包含 `libsmbclient` 协议：

- `_ff_libsmbclient_protocol`
- `libsmbc`
- `enable-libsmbclient`
- `enable-protocol=libsmbclient`

可复现脚本：

```bash
zsh docs/smb_poc_check.sh
```

脚本检查的是：

- `Libavformat.xcframework/macos-arm64_x86_64/Libavformat.framework/Libavformat`

含义：

- 当前播放器底层“有机会”直接播放 `smb://` 资源；
- 但这不等于应用层已具备可上线的 SMB 媒体库能力。

## 为什么“不可直接上线”

1. 扫描链路未具备 SMB 目录遍历实现
- 当前已上线的是 `local + webdav` 扫描。
- SMB 若要入库刮削，需要独立目录列举与文件属性读取能力（不能只靠播放器 `loadfile`）。

2. 认证/会话模型复杂于 WebDAV
- SMB 需要处理：域、来宾、NTLM 变体、会话重连、超时、权限失败重试。
- 这部分目前没有应用层连接管理器。

3. 离线缓存策略尚未扩展到远程源
- 任务7已明确：远程源缓存暂禁用。
- SMB 上线后若无缓存策略，会出现交互断层（用户看到 SMB 源但缓存不可用）。

4. 可维护性与构建风险
- 若引入额外 SMB 客户端库（例如 libsmb2 封装）会增加：
  - Xcode 集成复杂度
  - 多架构二进制体积与签名风险
  - 后续安全更新负担

## 方案对比

### 方案 A：沿用现有底层，先做 SMB 播放试点
- 做法：只在播放时构建 `smb://` URL 给 mpv。
- 优点：改动小，验证快。
- 缺点：没有扫描入库能力，不满足“媒体库直连”核心目标。
- 适用：内部验证，不建议直接对外发布。

### 方案 B：引入应用层 SMB 扫描客户端（推荐二期方向）
- 做法：新增 SMB scanner（对齐 WebDAV scanner 抽象）。
- 优点：功能闭环（扫描、刮削、播放路径统一）。
- 缺点：工程量大，依赖与维护成本高。

### 方案 C：继续 WebDAV 主路径（当前推荐）
- 做法：SMB 只用于发现提示；用户启用 NAS WebDAV 后接入。
- 优点：当前代码已落地，稳定性与迭代速度最好。
- 缺点：依赖 NAS 端开启 WebDAV。

## 上线建议

1. 当前版本：不发布 SMB 直连入口（继续提示“请启用 WebDAV”）。
2. 二期立项前置门槛：
- 明确 SMB 客户端选型（自带底层能力 vs 新增库）
- 完成最小 PoC：目录列举 + 文件 metadata + 认证重连
- 通过回归：扫描、刮削、播放、断线重连、错误提示
3. 通过门槛后再做 SMB 产品化开关。
